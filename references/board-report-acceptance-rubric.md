# Board Report Acceptance Rubric

**Version:** 1.0  
**Date:** 28 May 2026  
**Applies to:** CPO Board Report (Stage 9 output) — all audiences (board, RemCo, leadership)  
**Remediation item:** Fix 5 (P1) — AiQ Evidence Pack Remediation Brief, 28 May 2026

---

## Purpose

This rubric defines the minimum acceptance standard for a board report generated by the AiQ platform. "Populated = pass" is not sufficient. A report passes only when it meets all criteria below. Human sign-off is required before any report is circulated externally.

The rubric is enforced at two levels:

1. **Automated gate** — a `validateBoardReportRubric()` function runs on the assembled section map before the report is marked ready for export. Any section that fails the structural checks blocks the export with a clear error message.
2. **Human review** — a reviewer works through the checklist below and records sign-off in the evidence pack.

---

## Section-Level Structural Criteria

Each of the 6 sections must meet the following minimum requirements to pass the automated gate.

| Section | ID | Min words | Required content signals |
|---|---|---|---|
| Context & Mandate | `context` | 120 | Must reference the organisation name; must not be a generic template |
| Strategic Direction | `strategic_direction` | 150 | Must reference the strategy archetype or strategy statement |
| Initiative Portfolio | `initiative_portfolio` | 180 | Must name at least 2 specific initiatives from the portfolio |
| Investment Case | `investment_case` | 150 | Must reference at least one financial figure (£, cost, value, saving, ROI) |
| Capability Readiness | `capability_readiness` | 120 | Must reference at least one capability dimension or gap |
| Governance & Next Steps | `governance` | 100 | Must reference a next step, approval, or review cadence |

## Profile Figure Citation Criteria (Human Review)

The narrative must reference the client's key profile figures. A reviewer checks each of the following:

| Figure | Criterion | Pass / Fail |
|---|---|---|
| Headcount | Headcount figure (or a close approximation) appears in at least one section | |
| Revenue | Revenue figure (or a close approximation) appears in at least one section | |
| Payroll / HR cost | Payroll or HR cost figure appears in at least one section | |
| Organisation name | Organisation name appears in every section | |

**Note:** For DFS specifically, the authoritative figures are DFS-provided. Public filings (LSE/market data) are used only as a sanity-check floor. See `shared/dfsProfileConstants.ts` for the recorded values and their sources.

---

## Content Quality Criteria (Human Review)

These criteria cannot be automated and require a human reviewer to assess.

| Criterion | Standard | Pass / Fail |
|---|---|---|
| No generic filler | No section reads as a generic template that could apply to any organisation | |
| Initiative specificity | The initiative portfolio section names specific initiatives and explains their coherence as a portfolio, not a list | |
| Financial grounding | The investment case cites specific cost and value figures from the dashboard, not vague ranges | |
| Plain English | No jargon, no bullet points in the narrative body, no vocabulary blacklist violations | |
| Audience calibration | Tone and depth are appropriate for the selected audience (board / RemCo / leadership) | |
| Consistency | Figures and claims are consistent across sections (no contradictions) | |

---

## Automated Gate — Implementation

The `validateBoardReportRubric()` function in `server/boardReportRubric.ts` implements the structural checks. It is called:

1. Before the PDF export endpoint serves the file (`/api/pdf/board_report`)
2. Before the DOCX export endpoint serves the file (`/api/export/board-report-docx`)

If any section fails, the export is blocked and the response includes a structured error listing which sections failed and why.
